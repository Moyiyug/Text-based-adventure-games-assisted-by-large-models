> **依据**：严格对齐 `ChatsDocs/RULES.md`（下称 **RULES**）。  
> **范围**：在**不改变** RULES §5 硬性约束的前提下，对「RAG + 多轮对话 + 流式叙事 + META 协议」方向的后续工作做**优先级排序**与**决策前审问**。  
> **关联规范**：RULES §6 索引的 `BACKEND_STRUCTURE.md` §4.4（协议）、`APP_FLOW.md` §3.5.2（展示）、`PRD.md`（成功标准/范围）、`IMPLEMENTATION_PLAN.md`（实施节奏）。

## 1. 不可退让的约束（排序前提）

以下来自 **RULES §5**，任何优化方案若与之冲突，须**先改 RULES 与相关 docs 并达成共识**，再动代码：

| # | 约束 | 对优化方案的含义 |
|---|------|------------------|
| 1 | 游玩端 SSE 须 **`fetch + ReadableStream`**，禁用 `EventSource` | 服务端侧脚本/验收工具亦应保持「POST + 流式读体」语义一致（RULES §5.1）；优化焦点在**协议与后端切分**，不在更换浏览器传输 API。 |
| 2 | 叙事与 META 使用 **`\n---META---\n` 协议**（见 `BACKEND_STRUCTURE.md` §4.4） | 「分步 LLM」若改变产出形态，仍须能**无损映射**到落库结构（正文 + `metadata`），或先修订协议文档。 |
| 3 | 游玩页主气泡 **不得**默认把原始 META JSON 当正文（`APP_FLOW.md` §3.5.2） | 自动化测试应区分：**玩家可见泄漏** vs **SSE 原始 token 缓冲区形态**；后者严判可能与服务端 withhold 策略不一致，需产品定义「何为失败」。 |
| 4 | 密钥仅 `.env` + `pydantic-settings` | 新增开关/评测任务继续走环境变量与配置类，禁止硬编码。 |
| 5 | DB 变更须 Alembic | 若优化涉及会话事件、RAG 命中日志、多阶段调用元数据等持久化，必须迁移而非手改库。 |

## 2. 问题陈述（为何需要本文件）

近期讨论中已收敛几类现象（与代码现状可对照 `lessons.md`）：

1. **多轮 soak / 人工体验**：约在 **第 5～6 轮** 前后，模型更易在**单轮一体生成**中违约（META/JSON 混入叙事、流式 token 与解析边界抖动）。  
2. **分调用策略**：叙事、选项、系统信息**分开调用**时，主观质量往往更稳，但 **延迟与成本**上升。  
3. **RAG 一致性疑虑**：担心「分开投喂」后，各部分是否仍**锚定同一检索证据**；以及检索 **query 仅用玩家当前句** 在长对话中是否仍足够。

本文件将上述方向**拆解为可排期项**，并用 **审问清单** 迫使需求在具体指标上落地，避免「优化」无验收标准。

## 3. 优先级方案（建议默认顺序）

说明：**P0** = 不解决则持续违背产品底线或放大幻觉/泄漏；**P1** = 明显改善质量或可观测性；**P2** = 体验 polish 或研究性项。  
顺序假设：**低成本模型 + 桌面端游玩**（RULES §1）不变。

### P0 — 协议可信与「玩家可见」底线

| ID | 项 | 与 RULES 的锚点 | 预期收益 |
|----|----|-----------------|----------|
| P0-1 | **统一「何为泄漏」的验收定义**：落库正文、`metadata`、`SSE token` 三层分别允许什么 | RULES §5.2、`APP_FLOW.md` §3.5.2 | soak/评测与产品一致，减少误报与漏报。 |
| P0-2 | **流式路径下 META withhold / finalize** 与落库 `strip_*` 链路的**单测 + 集成阈值**（含多轮会话 fixture） | `BACKEND_STRUCTURE.md` §4.4 | 降低第 N 轮边界崩溃概率。 |
| P0-3 | **选项来源可观测**：`choices_source`、是否经 refine、是否 fallback 等写入 metadata（已有字段则补全文档） | 管理端调试叙事（RULES §5.2） | admin 可解释「这一轮选项为何长这样」。 |

### P1 — RAG 与「分步调用」对齐（架构债偿还）

| ID | 项 | 与 RULES 的锚点 | 预期收益 |
|----|----|-----------------|----------|
| P1-1 | **选项补洞（fallback）与主流程共用 `assemble_context` 节选**：调用方传入证据块（能力已在 `choice_fallback` 预留） | RULES §1「低幻觉」、`PRD.md` 成功标准 | 减少无证据的「纯叙事臆造选项」。 |
| P1-2 | **接入或删除** `choice_synthesis_rag.synthesize_choices_with_rag_context`：与主轮**同一 `context`** 的第二次调用须在 `engine` 中有明确分支；若弃用则删死代码并更新 `config` 注释 | `BACKEND_STRUCTURE.md` §4 管线 | 消除「文档/配置描述」与真实路径不一致。 |
| P1-3 | **检索 query 策略实验**（仍通过 `RagConfig` / 管理端可配置）：例如「玩家句 + 截断 GM 上一条」vs 现状；**不**默认增加移动端（RULES §1） | `TECH_STACK.md`、RAG 变体文档 | 改善中后期回合证据漂移，而不破坏桌面优先。 |
| P1-4 | **双阶段（`NARRATIVE_TWO_PHASE_*`）与 split-choices 开关**的产品说明：互斥关系、延迟预算、何时启用 | `IMPLEMENTATION_PLAN.md`、`.env.example` | 降低误配置导致的隐性单轮三调用等。 |

### P2 — 成本、延迟与评测体系

| ID | 项 | 与 RULES 的锚点 | 预期收益 |
|----|----|-----------------|----------|
| P2-1 | **分层 soak**：冒烟（无 LLM）/ 单轮 LLM / 多轮预算上限；与 `PRD.md` §7 对齐 | RULES §7 工作习惯、`progress.txt` | CI 与人类回归成本可控。 |
| P2-2 | **会话级 RAG 日志**（可选表或事件）：每轮 `query`、variant、top-k 摘要（注意勿存密钥） | 可观测性；密钥仍仅 `.env` | 复盘「为何检索偏了」。 |
| P2-3 | **前端**仅当 RULES §5.1 不变前提下优化感知延迟（骨架屏、流式节流），不引入 `EventSource` | `FRONTEND_GUIDELINES.md` | 体感优化，不改变安全与鉴权模型。 |

## 4. 审问清单（在启动每一项前须回答）

> 本节用于**审问具体想法**：未回答前，不建议把对应项标为「已承诺交付」。

### 4.1 产品：你愿意为「稳」付多少代价？

- **延迟**：单轮可接受的上限 P95 是多少秒？双阶段 = 至少多一次非流式 DeepSeek 往返，是否接受 **全员默认开启** 还是仅 strict 模式？  
- **成本**：每会话目标 token 上限或「每用户每日」预算是否有硬顶？  
- **失败策略**：若第二轮 META/选项调用失败，是 **重试**、**降级为单轮协议**，还是 **硬错误**？玩家看到哪一句 copy？

### 4.2 RAG：你要「锚定作品」还是「锚定当前剧情」？

- 检索 query **仅玩家当前句**（现状）是否在 PRD 意义上已足够？若否，**剧情摘要**由谁生成（同一模型另调用 / 规则截断 / 状态机字段）？  
- **第二次检索**（选项专用）若引入：是否与第一次检索 **合并为一次 hybrid query**，还是 **两次检索各取 top-k** 再拼接？边界情况（极短玩家句）如何处理？  
- **无命中/低相似度**时，是 **拒答**、**放宽阈值**，还是 **允许纯模型续写**？哪一种更符合「低幻觉」叙述？

### 4.3 协议与测试：什么算「绿」？

- soak 脚本中 **「stream token 含 `"choices"`」** 是否一律视为产品缺陷，还是 **仅当落库正文或玩家可见流式展示含 JSON** 才算？请二选一或写清分层。  
- **管理员调试折叠**（RULES §5.2）是否足以替代「向 player 暴露部分协议」的需求？若玩家也要「开发者模式」，是否违反当前范围？

### 4.4 范围：什么是明确不做？

- RULES §1：**不做移动端**——任何「为了手机端体验」的优化是否一律 P2 以下或直接拒绝？  
- **不做图像**——若未来要把选项做成「卡片配图」，是否必须先修订 PRD/RULES？

### 4.5 文档与流程：谁维护真相？

- 优化落地后，**必须**同步：`BACKEND_STRUCTURE.md`（管线）、`.env.example`、`lessons.md`（踩坑）、必要时 `RULES.md`（若约束变化）。  
- 是否指定 **唯一负责人** 审批「配置开关默认值」变更（避免生产环境与文档漂移）？
